Constraints & Conceptual Models

01Constraints: guide, don't block

A constraint is anything that reduces the set of actions a user can take. Done well, it does the work a signifier would — it makes the right move obvious without a label, because the wrong moves are quietly off the table. — Norman, DOET (2013), Ch. 4

"Constraints are powerful clues, limiting the set of possible actions. The thoughtful use of constraints in design lets people readily determine the proper course of action, even in a novel situation."
— Don Norman, The Design of Everyday Things

02The four constraints

The confusion to unlearn: a constraint is not the same as a restriction. A restriction blocks a user who is trying to act and reports the failure afterward. A good constraint guides — it removes the invalid option before the user reaches for it, so the error never forms. Norman names four kinds, strongest to most learned: — NN/G, "Constraints"

  • Physical — the form of the control makes the wrong input impossible. A slider bounded 0–100; a number field that won't accept letters; a date picker that can't express "Feb 31."
  • Logical — the user reasons out the valid move from the situation. "Continue" disabled until a choice is made; step 3 locked until step 2 is done; one empty slot, one piece left.
  • Semantic — the meaning of the thing narrows its use. A trash-can icon means delete; an envelope means message. You know what to do because you know what it is.
  • Cultural — an arbitrary convention shared by a group. Red means danger or stop; the hamburger means menu; an X closes. Nothing about red requires it — your audience just learned it.

Version B's dropdowns never offer a 30th of February at all: the invalid date is unexpressible, so the mistake can't happen. Same outcome, but one prevents the error where the other only reports it. 04Conceptual & mental models

A conceptual model is a simplified explanation of how something works. Two of them sit in every interaction. The system model is how the product actually behaves; the user's mental model is what the user believes it does. The user can't see your code or your design intent; they infer their model entirely from what the interface shows. Norman calls that visible surface the system image. — Norman, DOET, Ch. 1; NN/G, "Mental Models"

When the two models match, behavior is predictable and the user feels in control. When they diverge, the errors aren't random — they're systematic: the same wrong expectation fires every time, because the user is reasoning correctly from a model the interface taught them. You fix it by correcting the system image, not the user.

07Clinic: the sync toggle your turn

A file inside a folder that syncs to the cloud, as shipped. You're low on disk space and want this file off your laptop. Use the control that looks right.

free up space — get "Q3 Forecast.xlsx" off this device.
Synced folder · Work
Q3 Forecast.xlsx
What, if anything, is wrong here?
In a "synced folder," most people's mental model of "Delete from folder" is "remove the local copy, keep it in the cloud." The system model is the opposite: it deletes everywhere, on every device. The models diverge, so the error is systematic — anyone reaching to save space wipes the only copy. This is a conceptual-model mismatch, and the fix is to correct the system image.
The fix that closes the gap?
The user's intent ("keep it, just not here") was never expressible, so the model collided with reality. Naming both scopes makes the real system model visible and lets the user pick the action they actually meant. A spinner or a bigger button changes how the wrong action feels, not what the user believes it does.
Principle: a model mismatch is fixed in the system image (the labels, scopes, and structure the interface exposes), not in feedback or emphasis. When the user's reasonable belief diverges from the truth, change what the interface communicates until the belief is correct. You can't out-confirm a wrong mental model; you have to teach the right one.

10The expert's nuance

Over-constraining frustrates the people who know what they're doing

A constraint that protects a novice can cage an expert. Force-stepped wizards, fields that reject paste, modals that re-ask the obvious — useful the first time, friction the hundredth. Scaffold for the new user, then let the experienced one opt out: remembered defaults, a "skip setup," paste that's tolerated and validated rather than blocked.

Mental models are borrowed, so honor conventions unless the payoff is real

Users arrive with a model assembled from every other app they've used — that's what cultural constraints lean on. A trash icon, a left-swipe to delete, a pull-to-refresh: pre-installed in the user's head. Inventing your own forces re-learning and breaks the borrowed model. Break a convention only when the gain clearly beats the cost of re-teaching it.

You can only ever fix the system image

You never touch the user's mind directly. Norman's chain runs designer's model → system image → user's model, and the middle term is the only one you control. When a model mismatch shows up, the lever is always the same: change what the interface shows (labels, scope, structure, feedback) until the model it teaches is the true one.

0 words
Model answer: "Constraint cost: stripping inline hints removes the constraints that prevent invalid input. Errors no longer fail early and locally — they pile up at submit, across the gulf of execution, where the user has to read, locate, and re-fix each one. Prevention turns back into correction. Mental-model cost: with nothing guiding input, the user can't form a model of what's valid until the form rejects them; they're reasoning in the dark, and the rejection feels arbitrary rather than learnable. The grain of truth: inline validation is noise when it nags too early — firing a red error on an email before you've finished typing it. Timing resolves it: validate at the right moment (on blur, or once a field looks complete), show requirements up front as quiet hints, and confirm success inline. That keeps the constraint and kills the noise." Check your own answer for three things: you framed the loss as constraints moving from prevention to late correction, you named the mental-model gap, and you resolved the noise objection with timing rather than removal.

13Strengthen the muscle

Turn the reasoning into a conscious procedure. Protocol → guided walk-through → real-world mission.

Part A — The protocol (run it on any control)

Name the action and its valid set. What is the user doing here, and what counts as a valid input or move?
Constraint check (before the error). Is an invalid action hard or impossible to take? If yes, which type carries it — physical, logical, semantic, cultural?
Model check (after the action). What does the user believe this does, and what does it actually do? List both models; mark where they diverge.
Locate the gap. A constraint gap = invalid actions are possible or caught too late. A model gap = the behavior surprises the user the same way every time.
Prescribe to the right gap. Constraint → add or strengthen it, prefer prevention over an error message, pick the type that fits. Model → fix the system image: labels, scope, structure, feedback. Don't paper a model mismatch with a tooltip; don't fix a missing constraint by retraining the user.
