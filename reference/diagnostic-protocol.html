Affordance Diagnostic Protocol

Run this on any element until it runs itself. Change perception, not function.

The five steps

Name the affordance. What can this element actually do? Mentally strip the styling — what's the real capability underneath?
Find the signifier. What perceptible cue communicates it? If you can't point at one, there isn't one.
Place it on the 2×2. Affordance present? × signifier present? → one of four verdicts.
If broken, branch. Hidden or false? Then ask the branch questions below.
Prescribe. Add a signifier, remove a false one, or add a discoverable path. Rarely rebuild the behaviour.

The 2×2 (step 3)

Signifier absent

The branch (step 4)

Hidden → Is the action critical? Is there a discoverable equivalent? Only critical + no equivalent is a real defect. Otherwise it's a fine accelerator (shortcuts, swipes, long-press).

False → Which interaction signifier is lying — pointer cursor, hover-lift/motion, underline, button shadow? Strip that specific cue; keep honest static decoration.

The one rule that overrides "signify everything": no critical action may be reachable only by a hidden affordance. The target is discoverability, not total exposure. And signifier strength should match the action's stakes.
