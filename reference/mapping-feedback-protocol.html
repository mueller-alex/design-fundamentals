Reference Card · Lesson 02

Mapping & Feedback Protocol

Bridge both gulfs. Mapping before the action, feedback after — and never patch one with the other's tool.

The two gulfs

Before the actionGulf of execution"How do I do this? Which control?" → bridged by mapping.
After the actionGulf of evaluation"Did it work? What state am I in?" → bridged by feedback.

The five steps

Find the action cycle. A goal → a do → a check. Locate both gulfs in the interaction.
Mapping check (before). Can the user predict which control produces which effect, without a label? Natural = spatial layout or trusted convention.
Feedback check (after). Does the system confirm it heard, show what it's doing, and report the result? Immediate · informative · appropriate.
Time the feedback. Match the signal to the latency (table below).
Prescribe to the right gulf. Mapping → fix layout / convention. Feedback → add the right signal at the right time. A toast can't fix a layout; a layout can't fix a missing confirmation.

Feedback by latency (Nielsen's limits)

DurationWhat the user feelsMinimum feedback
< 0.1 sInstantaneousJust show the result
0.1–1 sNoticed, but flow intactUsually none / subtle cue
1–10 sWaiting, attention heldSpinner / busy state
> 10 sAttention breaksProgress bar + estimate; let them work
Watch-outs: over-feedback is a failure too — match intensity to importance (passive for routine, assertive only for consequential). Cultural mappings (red = stop, up = on) are local; verify they travel. For high-confidence actions, optimistic feedback (show success now, reconcile quietly) makes perceived latency zero — if you handle the rare failure gracefully.
