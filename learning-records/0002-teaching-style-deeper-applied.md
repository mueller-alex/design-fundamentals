# Feedback: lessons must be deeper, longer, and built on applied problem-solving

After Lesson 01, the learner asked for (a) more depth and length — he reads fast and a ~1.5-screen
lesson "won't sink in" — and (b) a problem-solving framing: *"how would you solve this problem knowing
this fundamental?"* with worked examples, plus more interactive examples per lesson.

**Why it matters:** The default "short lesson, small working memory" heuristic under-serves this learner.
He's an 11-year expert; recognition-level content is too thin. The fix is not just more prose (that would
blow working memory) but more *active* surface area: multiple diagnose-this / build-this / classify-this
widgets so length comes from doing, not reading.

**How to apply going forward:**
- Each lesson should contain several interactive challenges, at least some framed as "here's a broken
  interface / real scenario — diagnose it and choose the fix using today's principle."
- Aim for 3-5+ screens of fast-read content, chunked into active widgets.
- Keep the wrong/right demo, but add applied problem-solving and build-it exercises on top.
